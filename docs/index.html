Leads Management: a six-hour build

A lead-intake platform for a law practice. This page documents scope, build method, agent usage, tests, and results.

FastAPI, PostgreSQL, Next.js. Co-created with Claude Code. Repository: github.com/sriddhi/leads-app

DecisionChoice in this buildNext step

Intake fieldsRequired name, email, resume. Added phone and messageStructured visa-type selection
Same phone/email/nameLinked and flagged, not merged, not rejectedAgent-on-behalf re-inquiry workflow

The intake requirement was kept as specified. A free-text message was added because intake without it was incomplete. A structured visa-type field was out of scope but is the change that would make routing efficient rather than uniform. A repeat of the same phone, email, and name can be a valid new inquiry, for example submitted by an agent on someone's behalf; that workflow was out of scope, so the system links and flags rather than merging, which keeps it open later.

Scope: covered, not covered, next

Covered

  • Public form: name, email, required resume (PDF/DOC/DOCX), plus phone and message
  • Email to prospect and attorney on submit; working interface, viewable in MailHog
  • Auth-guarded internal UI listing leads
  • Status PENDING to REACHED_OUT, set by the attorney, enforced server-side
  • Storage, email service, production-style structure, public repo, one-command run
  • Design doc, architecture doc, runbook, agent-usage doc, commit attribution
  • Added: admin dashboard, queue and assignment, capacity, audit trail, per-state time tracking, duplicate link-and-flag, case history, retention cleanup, load recipe, AWS deployment design (no implementation)

Not covered (deliberate)

  • Structured visa-type selection and skill-based routing; only a free-text message exists
  • Multiple attorneys per case or collaboration
  • Agent-on-behalf re-inquiry as an intentional new-case workflow
  • MFA and advanced auth; backend is structured to accept it, not enabled
  • Real email provider and object storage in production; MailHog and local disk stand in
  • Multi-replica scale; rate limiter is in-memory, storage is local by design

Next

  • Visa-type field, route on specialization instead of uniform capacity
  • S3 storage, SES email, shared rate limiter, multi-AZ RDS, autoscaling ECS; the seams exist
  • Maintained capacity counter or SELECT FOR UPDATE SKIP LOCKED on the assign hot path
  • Agent-on-behalf re-inquiry on top of link-and-flag
  • Observability and alarms; MFA as an option

Architecture

Stack: FastAPI (async), PostgreSQL 16, SQLAlchemy and Alembic, JWT auth, Next.js 14, Tailwind, Docker Compose. Layering is one-directional: thin endpoints, business logic in services (pure where possible), data access in CRUD, Postgres as the source of truth.

Correctness mechanisms

MechanismWhat it guarantees
Optimistic lockingEach lead has a version. A mutating write checks the expected version and increments it. Concurrent assignment yields one owner and a 409 for the other. No lost updates.
Append-only auditEvery state change writes an immutable row with actor, action, before, after. No update or delete path. Feeds a live SSE stream.
Interval timelineEach state a lead enters is stored with entered_at and exited_at. Exactly one period open per lead. Attorney holding time is a query, not an estimate.
Duplicate link-and-flagMatches on normalized email or phone are linked and flagged, never merged. Every application stays a separate, auditable record.

Boundary hardening

Honeypot, per-IP rate limiting, idempotency keys, streaming upload-size cap, content-based file validation by magic bytes, and a minimal public receipt that exposes no internal fields. Storage and email are pluggable, so the AWS migration is configuration, not a rewrite.

Building with the agent

Claude Code was used as an orchestrator. The division of labor was the deciding factor.

Owned by handDelegated to the agent
Data model, concurrency strategy, threat model, AWS seams, and the specifications: exact field names, paths, status codes, invariants. Implementation against fixed specs: models, migrations, schemas, endpoints, services, the typed frontend, containers, first drafts of docs and tests.

Verification was non-negotiable: run it. Reading agent output shows it is plausible; executing it shows it is correct.

Runtime defect example

The agent pinned passlib[bcrypt]==1.7.4 but left bcrypt unpinned. The code type-checked and reviewed clean. Resolution pulled a modern bcrypt; the old passlib runs an import-time probe that hashes an over-long string, and the new library raises instead of truncating. Every password hash threw, so seeding and all logins failed. The fault was in the interaction between two reasonable choices, not in any line. Found by running seed.py. Fix: pin bcrypt==4.0.1. A later defect, a record number that collided after deletions left gaps, was caught the same way, by the load test.

Operating principles applied

  • Think before coding: the test plan was contested twice before harness code existed.
  • Simplicity first: one database, no broker, no speculative abstractions.
  • Surgical changes: defects fixed with one-line, localized edits plus a regression test.
  • Goal-driven: work backward from invariants and the error budget, so "done" was measurable.

Agent effectiveness vs context size

Output quality is not uniform across a session. It falls as the context window fills. The first four hours were efficient; around hour five, effective output per turn dropped and the context required compaction.

Relative effective output per turn vs session hour. Approximate, qualitative.

Mechanism and remedy: the full history is reprocessed each turn, so per-turn cost and latency rise with length, and early constraints compete with everything since and get dropped. Keep context lean. Scope each session to one task, summarize, restart, and re-state the invariants that must survive.

Process, in eleven steps

  1. Basic use case: three roles, one intake-to-outreach path.
  2. Constraints: one attorney per case, equal empowerment, link-not-merge, size and field limits, threat model.
  3. MVP: thinnest slice that runs end to end across all roles.
  4. Structure and unit tests: layering and a pure, fast-to-test core.
  5. Fill the details: the flows the real system needs.
  6. One large implementation against the fixed specs.
  7. Integration test suite design: state-based invariants and an error budget.
  8. Email integration: pluggable backends, fire-and-forget, viewable locally.
  9. Run the integration tests; fix what running exposes.
  10. Human test run: every role through the UI by hand.
  11. Document and push: readme, design, architecture, runbook, agent usage; clean history.

Tests and results

The suite is built to be able to fail. Correctness is checked with state-based invariants, not timing, so results are deterministic regardless of host speed. A self-test plants known defects first to prove the harness can go red before any run is trusted.

Gates

  • Error budget: run fails if the request error rate exceeds the configured budget, set at 2 percent, adjustable per run
  • Zero server errors: any 5xx fails the run
  • All invariants hold: single assignee, capacity exact at the boundary, distinct leads (no merge), one open period per lead, audit completeness, history semantics, pinned public-receipt contract, retention

The recipe drives 1,000 applications through 75 concurrent attorneys at capacity 10, exercises every flow and edge case, and restores a clean baseline afterward. Source: backend/tests/load. Latest run:

1,000 / 1,000 created, zero errors 0.00% error rate (gate 2%) 0 server errors 100% invariants held

Under that load the system recorded thousands of version conflicts and zero double-assignments. The lock degraded into bounded contention, not corruption.

Agent critique

LimitationObserved in this build
Plausible but wrong in interactionsFiles reviewed clean while the system failed at runtime. The dependency-interaction bug and the gap-induced number collision both passed inspection. Only execution closes the gap.
Degrades as context growsQuality and efficiency fall as the window fills. Early constraints get dropped, work is repeated, per-turn cost climbs. The operator must ration context.
Expands scope unless held backAdds abstraction and surface area unprompted. Simplicity is a constraint to impose, not the default.

Net: the agent moved the bottleneck from typing to deciding and from writing to verifying. The leverage is real and front-loaded into the specification and the verification.

Run it

One prerequisite: Docker. The system migrates and seeds itself, including one worked example case, so there is nothing to configure.

git clone https://github.com/sriddhi/leads-app.git && cd leads-app
docker compose up --build
